# Proactive AI Tutor — Feature Documentation

## 1. Overview

The Proactive AI Tutor transforms IELTS Journey's AI Tutor from a passive chatbot into an active personal tutor that observes the learner's progress, understands their weaknesses, reminds them at the right time, and helps them know exactly what to do next.

Every proactive message is **data-driven** — never random or generic. Each message is traceable to a specific user-data trigger (due vocabulary reviews, mistake patterns, weak skills, exam countdown, study streak, inactivity, etc.).

### Key Principles

1. **Every message has a reason** — each message is traceable to a specific trigger and user data
2. **Priority-aware delivery** — messages ranked by urgency (high/medium/low), capped per day
3. **Annoyance prevention by default** — rate limits, quiet hours, cooldowns, category toggles, snooze/dismiss
4. **Multi-surface, single source** — unified message queue consumed by all surfaces
5. **Template + AI hybrid** — routine messages use templates; optional AI enhancement when API key is configured

## 3. Service Layer Architecture

### 3.1 ProactiveMessageEngine (`packages/ai-tutor/src/services/proactiveMessageEngine.ts`)

**The core message generation engine.** Contains 21 individual generator functions, each a pure function receiving a `ProactiveEngineInput` and returning a `ProactiveMessage` or `null`.

**Key API:**

```typescript
// Generate all possible messages (no filtering)
function generateProactiveMessages(input: ProactiveEngineInput): ProactiveMessage[]

// Generate with settings-aware filtering
function generateProactiveMessagesWithSettings(
  input: ProactiveEngineInput,
  settings: ProactiveMessageSettings,
  existingMessages: ProactiveMessage[],
): { messages: ProactiveMessage[]; throttled: number; reason?: string; delayUntil?: string }

// Gatekeeper checks
function canSendProactiveMessage(settings, existingMessages): CanSendResult
function isCurrentlyQuietHours(settings): boolean
function getNextAllowedTime(settings, existingMessages): string | undefined
```

**Engine pipeline:**

```
Timer tick / event trigger
       │
       ▼
 ┌─────────────┐
 │ Gatekeeper   │ ← checks: enabled, quiet hours, daily cap, cooldowns
 └──────┬──────┘
        │ (pass)
        ▼
 ┌──────────────┐
 │ Compute State│ ← builds ProactiveEngineInput from storage
 └──────┬───────┘
        │
        ▼
 ┌──────────────────┐
 │ Run all generators│ ← 21 pure functions
 └──────┬───────────┘
        │
        ▼
 ┌─────────────────┐
 │ Priority Sorter  │ ← sort by priority, deduplicate, apply cooldowns
 └──────┬──────────┘
        │
        ▼
 ┌──────────────────┐
 │ Capacity filter  │ ← maxMessagesPerDay check
 └──────┬───────────┘
        │
        ▼
 ┌──────────────────┐
 │ Apply tone       │ ← friendly/strict/motivational/simple/vietnamese
 └──────┬───────────┘
        │
        ▼
 ┌──────────────────┐
 │ Enqueue & Notify │ ← store, emit event via ProactiveEventBus
 └──────────────────┘
```

**Tone application:** Each message is optionally prefixed based on the configured tutor tone:
- `friendly`: "Hey there! " + message
- `motivational`: "Let's go! " + message
- `vietnamese`: "Chào bạn! " + message
- `strict`/`simple`: No prefix

**Cooldown enforcement:** Each trigger type has a minimum interval between repeats:

| Trigger Type | Cooldown |
|---|---|
| `due_review` | 4 hours |
| `study_streak` | 24 hours |
| `low_activity` | 12 hours |
| `weekly_review` | 6 days |
| `monthly_review` | 28 days |
| `daily_tip` | 24 hours |
| `progress_celebration` | 24 hours |
| `lesson_completed` | 2 hours |

### 3.3 ProactiveEventBus (`packages/ai-tutor/src/services/proactiveEventBus.ts`)

Simple pub/sub system for real-time cross-surface synchronization.

**Events:**

| Event | Payload | Emitted When |
|---|---|---|
| `newMessage` | `ProactiveMessage` | A new message is created and stored |
| `messageRead` | `id: string` | A message is marked as read |
| `messageDismissed` | `id: string` | A message is dismissed |
| `messageSnoozed` | `id: string, until: string` | A message is snoozed |
| `messagesCleared` | — | All messages are cleared |
| `settingsChanged` | `ProactiveMessageSettings` | Settings are updated |

Handlers return unsubscribe functions: `const unsub = ProactiveEventBus.onNewMessage(callback)`

## 6. Frequency & Quiet Hours Enforcement

### 6.1 Daily Cap

- Default: **5 messages per day** (configurable 1–50 in settings)
- Enforced in `canSendProactiveMessage()` — checked before any generation
- Counter resets at midnight
- When cap is reached: `getNextAllowedTime()` returns next midnight

### 6.2 Quiet Hours

- Default: **22:00 – 08:00** (configurable)
- Supports overnight ranges (e.g., 22:00–08:00 crosses midnight)
- `isInQuietHours()` checks current time against configured window
- `shouldDelayToQuietHours()` returns delay duration if currently in quiet hours
- Messages generated during quiet hours are queued for delivery when quiet hours end

### 6.3 Priority Preemption

If the daily cap is reached and a high-priority message arrives, it can replace the lowest-priority unread message (handled by the `generateProactiveMessagesWithSettings` capacity budget).

### 6.4 Category Toggles

Each of the 13 categories can be independently enabled/disabled. Disabled categories are skipped entirely during generation. Queued messages of a disabled category are not removed — the setting only applies to new generation.

### 6.5 Adaptive Frequency

Based on user interaction feedback (`generateAdjustmentSignals`):
- If ≥3 feedback entries and helpful ratio < 30% with ≥2 "too-frequent" dismissals → reduce frequency
- If ≥3 feedback entries and helpful ratio > 80% with no "too-frequent" dismissals → increase frequency
- Categories with repeated dismissals are flagged as blocked

## 7. User Feedback & Interaction Handling

### 7.1 User Actions

| Action | Effect | Event Emitted |
|---|---|---|
| **Dismiss** | `isDismissed = true`, `dismissedAt = now` | `emitMessageDismissed(id)` |
| **Snooze** | `isSnoozed = true`, `snoozedUntil = now + N` | `emitMessageSnoozed(id, until)` |
| **Action click** | `isRead = true`, `actionClicked = type` | `emitMessageRead(id)` |
| **Feedback** | `feedback = rating` | `emitMessageRead(id)` |
| **Read** | `isRead = true`, `readAt = now` | `emitMessageRead(id)` |

### 7.2 Interaction Stats

Aggregated stats available for adjusting future generation:
- `totalMessages`, `totalDismissed`, `totalSnoozed`, `totalActionClicks`, `totalFeedback`
- `helpfulCount`, `notHelpfulCount`
- `dismissReasonCounts` — breakdown by reason
- `actionClickCounts` — breakdown by action type
- `interactionRate` — ratio of messages with any interaction

### 7.3 Connected Message Listener

`useProactiveMessages()` hook provides:
- Real-time message list via `ProactiveEventBus.onNewMessage`
- Message management (read, dismiss, snooze, delete, clear)
- Settings management
- Per-category filtering
- Generation from engine input

### 8.2 Data Flow

```
User Action (lesson complete, mistake saved, etc.)
       │
       ▼
proactiveMessageTriggers.onXxx()
       │
       ├──▶ Loads data from localStorage
       ├──▶ Calls generateProactiveMessagesWithSettings()
       │        │
       │        ├──▶ Gatekeeper checks (enabled, quiet hours, daily cap)
       │        ├──▶ Runs 21 generator functions
       │        ├──▶ Filters by enabled categories
       │        ├──▶ Checks per-trigger cooldowns
       │        ├──▶ Sorts by priority
       │        └──▶ Applies tone
       │
       ├──▶ ProactiveMessageStorage.addMessage()
       └──▶ ProactiveEventBus.emitNewMessage()
                │
                ▼
       All subscribed surfaces re-render
```

### 8.3 Dependency Graph

```
proactiveMessageTypes (pure types + schemas)
    ↓
proactiveMessageEngine (generators, gatekeeper, tone)
    ↓
proactiveMessageTriggers (wraps engine + triggers)
    ├── proactiveMessageStorage (persistence)
    └── proactiveEventBus (pub/sub)
            ↓
    proactiveNotificationService (browser notifs)
    proactiveMessageInteraction (user actions)
    proactiveProgressReview (weekly/monthly reviews)
            ↓
    useProactiveMessages (React hook)
            ↓
    Dashboard/Chat/Extension UI components
```

## 10. Configuration Options

| Setting | Type | Default | Description |
|---|---|---|---|
| `enabled` | boolean | `true` | Master toggle |
| `tone` | enum | `'friendly'` | Tutor personality: friendly, strict, motivational, simple, vietnamese |
| `preferredStudyTime` | string (HH:mm) | `'09:00'` | Preferred time of day for studying |
| `dailyReminderTime` | string (HH:mm) | `'09:00'` | When to send the first daily reminder |
| `reminderFrequency` | enum | `'smart'` | daily, every-other-day, weekly, smart |
| `weakSkillPriority` | string[] | `[]` | Ordered list of skills to prioritize |
| `notificationChannels` | enum[] | `['in-app']` | in-app, browser, extension |
| `automationLevel` | enum | `'semi-automatic'` | manual, semi-automatic, automatic |
| `autoSuggestExercises` | boolean | `true` | Auto-generate exercise suggestions |
| `autoWeeklyReview` | boolean | `true` | Auto-generate weekly progress reviews |
| `categories` | Record | all `true` | Per-category enable/disable |
| `quietHoursStart` | string (HH:mm) | `'22:00'` | Quiet hours start time |
| `quietHoursEnd` | string (HH:mm) | `'08:00'` | Quiet hours end time |
| `maxMessagesPerDay` | number (1–50) | `5` | Daily message cap |

### Recommended Configuration for Different Users

**New learner (just started):**
```json
{
  "enabled": true,
  "tone": "friendly",
  "reminderFrequency": "daily",
  "maxMessagesPerDay": 3,
  "categories": { "vocabulary-review": true, "motivation": true, "daily-tip": true }
}
```

**Active learner (preparing for exam):**
```json
{
  "enabled": true,
  "tone": "motivational",
  "automationLevel": "automatic",
  "autoSuggestExercises": true,
  "autoWeeklyReview": true,
  "maxMessagesPerDay": 7,
  "categories": { "exam-countdown": true, "mistake-review": true, "writing-practice": true }
}
```

**Vietnamese learner:**
```json
{
  "enabled": true,
  "tone": "vietnamese",
  "reminderFrequency": "smart",
  "notificationChannels": ["in-app", "browser"]
}
```

## 11. Developer Guidelines

### 11.1 Adding a New Generator

1. Define a new trigger type in `ProactiveMessageTriggerType` enum (`types/proactiveMessage.ts`)
2. Add a new category if needed in `ProactiveMessageCategory` enum
3. Create a generator function following the `MessageGenerator` signature:
   ```typescript
   function generateNewTrigger(input: ProactiveEngineInput): ProactiveMessage | null
   ```
4. Add the generator to the `generators` array in `proactiveMessageEngine.ts`
5. Add it to `GENERATORS` export map
6. Add an action preset in `CATEGORY_ACTIONS` if needed
7. Add trigger cooldown in `TRIGGER_COOLDOWN_MS` if needed
8. If trigger should fire from a user action, create a trigger function in `proactiveMessageTriggers.ts`

### 11.3 Adding to a New Surface

1. Subscribe to `ProactiveEventBus.onNewMessage(callback)` to receive real-time updates
2. Use `ProactiveMessageStorage.loadMessages()` to get the current state
3. Filter/display messages as appropriate for the surface
4. Wire dismiss/snooze/read actions back through `ProactiveMessageInteraction`
5. For React surfaces, use `useProactiveMessages()` hook

## 12. Anti-Annoyance Summary

| Mechanism | Implementation |
|---|---|
| **Daily cap** | Configurable (default 5), enforced before generation |
| **Quiet hours** | Configurable (default 22:00–08:00), blocks all delivery |
| **Per-trigger cooldown** | Minimum interval per trigger type (2h–28d) |
| **Category toggles** | Users disable categories they don't want |
| **Dismiss** | Permanently removes message from all surfaces |
| **Snooze** | Hides for configurable duration (15m–1 week) |
| **Tone selection** | 5 tone options matching user preference |
| **Reminder frequency** | daily/every-other-day/weekly/smart |
| **Adaptive frequency** | Based on dismiss/feedback patterns |
| **Expiry** | Messages auto-expire after configured TTL |
| **Empty state hidden** | No UI shown when no messages are pending |
| **Max per trigger type** | Same trigger cannot appear twice in the same day |

## 13. Edge Cases

| Edge Case | Behavior |
|---|---|
| User disables proactive tutor | All pending messages hidden. No new messages generated. Engine exits early at gatekeeper. |
| User switches tone mid-day | Next generated messages use new tone. Already-queued messages keep original tone. |
| User changes quiet hours to overlap all day | No messages delivered. Generation is paused. |
| User never opens settings | Defaults apply: enabled, 5/day, friendly, all categories on. |
| All categories disabled | No messages generated. Engine skips all triggers. |
| localStorage full | Messages not saved; `safeJsonSet` catches silently. |
| Daily cap reached + high-priority event | Priority preemption evicts lowest-priority unread message. |
| Extension and web app both open | Both read/write same localStorage — synchronized via EventBus. |
| No API key configured | AI enhancement disabled. Template messages used instead. |
| First launch | No existing settings; defaults applied. All triggers ready to fire once data exists. |
